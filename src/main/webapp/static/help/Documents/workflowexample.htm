You Are Here: Kuali Enterprise Workflow (KEW)

Workflow Example

In the following example, a user submits a Proposal Development document for approval to www.grants.gov.  The institution’s business rules state the institutional personnel must approve the proposal in the following order:

1.  Proposal Key Personnel – all key personnel must approve

2.  Department Head – the department head must approve

3.  OSP – anyone in OSP may approve

Figure 410 Example Proposal Development Workflow Submission

 

In the example, there are three nodes:  Proposal Key Personnel, Department Head, and OSP.  Each node must complete their approvals before KC routes the document to the next node.

Proposal key personnel may approve concurrently.  They can approve in any order but they must complete all of their approvals before KC routes the document to the department head.

The department head has designated a delegate.  If a delegator designates a primary delegate, KC routes the action request to the primary delegate.  The primary delegate must complete the action request for KC to proceed sending the document to the next node in the workflow.  If the delegator designates a secondary delegate, KC routes the action request to both the delegator and the secondary delegate.  Either the delegator or the delegate can complete the action request for KC to proceed with the workflow.

OSP is a workgroup in KC.  Only one member of a workgroup must approve the document for KC to send it to the next node.  After any OSP staff member approves the document, KC marks it as complete for submission to www.grants.gov.

When the document is eligible for submission to www.grants.gov, the OSP administrator initiates the system-to-system (S2S) submission.  When the S2S submission completes, KC marks this workflow complete.

Authorized users and approvers may add ad hoc approvers to the workflow.  Ad hoc approvers may approve the proposal concurrent to the key personnel, department head and OSP approvals.
